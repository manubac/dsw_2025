# Store Registration — Design Spec
**Date:** 2026-04-29  
**Status:** Approved

## Overview

A separate, invite-only registration page for vendedores/tiendas. A tienda receives a single-use link (generated by an admin endpoint); they fill out their data and verify email + phone (WhatsApp) before the account is created. The page is always live at `/register-store` but does nothing without a valid, unused invite token in the URL.

---

## Architecture

### Approach chosen
Token-anchored verification state (Approach A): the `StoreInvite` DB record tracks `emailVerified` and `phoneVerified` as booleans. Each verification step updates the record. The final `/complete` step checks both flags before creating the User + Vendedor.

## Backend Detail

### `StoreInvite` entity

| Field | Type | Notes |
|---|---|---|
| `id` | int (BaseEntity) | PK auto |
| `token` | string, unique | UUID v4 |
| `used` | boolean, default false | set true on `/complete` |
| `emailVerified` | boolean, default false | set true on `/verify-email` |
| `phoneVerified` | boolean, default false | set true on `/verify-phone` |
| `createdAt` | datetime (BaseEntity) | auto |

### Admin endpoint

`POST /api/admin/store-invite`  
- Requires header `x-admin-key` matching `process.env.ADMIN_SECRET ?? "admin123"`  
- Generates a UUID token, persists `StoreInvite`, returns:
  ```json
  { "link": "http://localhost:5173/register-store?token=<uuid>" }
  ```
- No expiry for now (tokens are valid until used)

### Store-register endpoints

All endpoints receive `token` and look up the `StoreInvite`. They return 400 if the token is missing, not found, or already `used`.

| Method | Path | Body | Action |
|---|---|---|---|
| GET | `/api/store-register/validate` | `?token=` | Returns `{ valid: true }` if invite exists and `!used` |
| POST | `/api/store-register/verify-email` | `{ token, code }` | Checks `code === "123456"`, sets `emailVerified=true` |
| POST | `/api/store-register/verify-phone` | `{ token, code }` | Checks `code === "123456"`, sets `phoneVerified=true` |
| POST | `/api/store-register/complete` | `{ token, nombreTienda, email, password, telefono, ciudad, direccion, piso, departamento, alias, cbu, descripcion }` | Validates `emailVerified && phoneVerified && !used`, creates `User` + `Vendedor`, marks `used=true`, returns JWT |

**`/complete` creates:**
- `User`: `username=nombreTienda`, `email`, `password` (bcrypt), `role="user"` (default — el rol "vendedor" lo determina el JWT dinámicamente al detectar que existe un `Vendedor` asociado), `is_email_verified=true`, `is_phone_verified=true`
- `Vendedor`: `user`, `nombre=nombreTienda`, `telefono`, `ciudad`, `direccion`, `piso`, `departamento`, `alias`, `cbu`, `descripcionCompra=descripcion`
- Returns same JWT shape as the existing seller upgrade flow: `{ token, data: vendedor }`

## Frontend Detail

### Route
Public, no auth guard:
```tsx
<Route path="register-store" element={<StoreRegistrationPage />} />
```

### `StoreRegistrationPage.tsx` — 4-step wizard

**Step 0 — Token validation (on mount):**
- Reads `?token=` from URL
- Calls `GET /api/store-register/validate?token=`
- If invalid or used: shows full-page error screen ("Este link de registro no es válido o ya fue utilizado")
- If valid: proceeds to Step 1

**Step 1 — Form data:**

Fields (all required unless noted):
- Nombre de tienda
- Email
- Confirmar email (client-side match validation)
- Contraseña (min 6 chars, strength indicator — reuse pattern from `UserRegistration.tsx`)
- Confirmar contraseña
- Teléfono (format `+54 9 XXXX XXXX`, same regex as seller upgrade)
- Localidad — uses the `CityPicker` component extracted from `UserRegistration.tsx`
- Dirección
- Piso (optional)
- Departamento (optional)
- Alias (optional)
- CBU (optional)
- Descripción (optional, textarea)

On "Continuar": client-side validation only, advances to Step 2.

**Step 2 — Email verification:**
- Banner: "Ingresá el código de verificación enviado a `<email>`"
- Note: "(En modo testing, el código es siempre 123456)"
- 6-digit input
- "Verificar" → `POST /api/store-register/verify-email { token, code }`
- On success: advance to Step 3

**Step 3 — Phone verification:**
- Banner: "Ingresá el código de verificación enviado por WhatsApp a `<telefono>`"
- Note: "(En modo testing, el código es siempre 123456)"
- 6-digit input
- "Verificar" → `POST /api/store-register/verify-phone { token, code }`
- On success: calls `POST /api/store-register/complete` with all form data + token

**Step 4 — Success:**
- "¡Tu cuenta de vendedor fue creada!"
- Auto-login via `login()` from user context + JWT from `/complete` response
- Button "Ir a mi perfil" → navigates to `/mi-perfil`

### Style
Matches `UserRegistration.tsx` visual style (white card, rounded-2xl, Tailwind, same primary color).

## Testing

Hardcoded verification codes: both email and phone codes are always `"123456"`. No actual emails or WhatsApp messages are sent.

Admin key for generating invites: `process.env.ADMIN_SECRET ?? "admin123"`.

To test end-to-end:
1. `POST /api/admin/store-invite` with header `x-admin-key: admin123` → get link
2. Open link in browser
3. Fill form, enter `123456` for both verifications
4. Confirm account is created and login works
